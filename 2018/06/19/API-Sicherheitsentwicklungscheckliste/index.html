API-Sicherheitsentwicklungscheckliste

Jun 19, 2018

I. Identitätsauthentifizierung

  • [ ] Verwenden Sie Basic Auth nicht, um Standardauthentifizierungsprotokolle (wie JWT, OAuth) zu verwenden.

  • [ ] Machen Sie nicht “Authentifizierung”, “Token generieren”, “Passwort speichern” diese Räder, verwenden Sie Standard.

  • [ ] Benutze Max Retry und blockiere automatisch bei der Anmeldung.

  • [ ] Verschlüsseln Sie alle sensiblen Daten.

1.1 JWT (JSON Web Token)

  • [ ] Verwenden Sie einen zufälligen und komplexen Schlüssel (JWT Secret), um die Schwierigkeit der Brute-Force zu erhöhen.

  • [ ] Daten nicht direkt im Anfragetext extrahieren und die Daten verschlüsseln (HS256 oderRS256).

  • [ ] Machen Sie die Ablaufzeit des Tokens so kurz wie möglich (TTL,RTTL).

  • [ ] Speichern Sie keine sensiblen Daten im JWT-Anfragetext, sondern [breakable] (https://jwt.io/#debugger-io).

1.2 OAuth-Autorisierungs- oder Authentifizierungsvereinbarung

  • [ ] Überprüfen Sie immer redirect_uri im Hintergrund, nur URLs auf der weißen Liste sind erlaubt.

  • [ ] Fügen Sie nicht jedes Mal, wenn das Token ausgetauscht wird, ein Token hinzu (nicht erlaubt: “response_type = token”).

  • [ ] Verwenden Sie den state -Parameter und füllen Sie einen zufälligen Hash, um Cross-Site-Request-Forgery (CSRF) zu verhindern.

  • [ ] Definieren Sie Standardbereiche und ihre jeweiligen gültigen Bereichsparameter für verschiedene Anwendungen.

☆, Besuch

  • [ ] Beschränken Sie den Datenverkehr, um DDoS-Angriffe und Brute-Force-Angriffe zu verhindern.

  • [ ] Verwenden Sie das HTTPS-Protokoll auf dem Server, um MITM-Angriffe zu verhindern.

  • [ ] Verwenden Sie das HSTS-Protokoll, um SSLStrip-Angriffe zu verhindern.

III, Eingabe

  • [ ] mit dem Betrieb der Betriebsfunktion HTTP konsistent verwendet, die GET ‚(Lesen),die POST (create),die PUT (ersetzen / update) unddie DELETE (Löschen von Datensätzen)Wenn das Verfahren nicht anwendbar ist Gibt die405 Method Not Allowed` für die angeforderte Ressource zurück.

  • [ ] Content-Feld in dem Baumuster zur Request-Header verwendet wird, nur das Content Format, um zu überprüfen (z as application / xml, application / jsonetc.) und zurückgegeben, wenn die Bedingung nicht 406 satisfied ist nicht Akzeptabel.

  • [ ] Datenvalidierung Content-Baumuster zur veröffentlichen und wie Sie erhalten (wie as application / x-www-form-urlencoded, multipart / form-data,application / json` etc.).

  • [ ] Verifiziere Benutzereingaben, um einige häufige Schwachstellen zu vermeiden (zB XSS,SQL-inject, ferne Codeausführung, etc.).

  • [ ] Sie keine sensiblen Daten ( credentials,Passwords, Sicherheit tokens, orAPI keys) in der URL, aber die Verwendung von Standard-Authentifizierungs-Request-Header.

  • [ ], um einen API-Gateway-Dienst zu ermöglichen, Caching, Zugriffsrate zu begrenzen (wie Quota,Spike Arrest, Concurrent Rate Limit) und dynamisch APIs Ressourcen bereitstellen.

IV, Verarbeitung

  • [ ] Überprüfen Sie, ob alle Terminals authentifiziert sind, um ein beschädigtes Authentifizierungssystem zu vermeiden.

  • [ ] Vermeiden Sie die Verwendung eindeutiger Ressourcen-IDs. Verwenden Sie / me / orders anstelle von/ user / 654321 / orders

  • [ ] Verwenden Sie UUID anstelle von Self-Growth ID.

  • [ ] Wenn Sie die XML-Datei syntaktisch analysieren müssen, stellen Sie sicher, dass die Analyse von Entitäten deaktiviert ist, um XXE-Angriffe zu vermeiden.

  • [ ] Wenn Sie die XML-Datei analysieren müssen, um sicherzustellen, dass die Unternehmen Expansion (entity Expansion), um geschlossen ist Entitätsexpansion Einheit Expansion Angriffe durch Index / XML bomb erreicht zu vermeiden.

  • [ ] Verwenden Sie CDN beim Hochladen von Dateien.

  • [ ] Wenn Sie eine große Datenmenge verarbeiten müssen, verwenden Sie Workers und Queues, um schnell zu reagieren und so HTTP-Blockierungen zu vermeiden.

  • [ ] Vergiss nicht, den DEBUG-Modus auszuschalten.

五, Ausgabe

  • [ ] Sende X-Content-Type-Optionen: nosniff-Header.

  • [ ] Sende X-Frame-Optionen: SAMEORIGIN (oder leugnet) header.

  • [ ] Sende `Content-Security-Policy: default-src ‘none’’-Header.

  • [ ] Lösche Fingerabdruck-Header - X-Powered-By,Server, X-AspNet-Version, etc.

  • [ ] Erzwingt content-type als Antwort, wenn Ihr Typapplication / json ist, dann ist Ihr content-type` application / json`.

  • [ ] Geben Sie keine vertraulichen Daten wie “Anmeldeinformationen”, “Kennwörter”, “Sicherheitstoken” zurück.

  • [ ] Gibt am Ende des Vorgangs den entsprechenden Statuscode zurück (z. B. “200 OK”, “400 ungültige Anforderung”, “401 nicht autorisiert”, “405 Methode nicht zulässig” usw.).

VI. Kontinuierliche Integration und kontinuierliche Bereitstellung

  • [ ] Verwenden Sie Komponententests und Integrationstests, um Ihr Design und Ihre Implementierung zu überprüfen.

  • [ ] Führen Sie den Code Review-Prozess ein, genehmigen Sie Änderungen nicht selbst.

  • [ ] Stellen Sie sicher, dass alle Komponenten des Dienstes statisch mit Antivirensoftware, einschließlich Bibliotheken von Drittanbietern und anderen Abhängigkeiten, gescannt werden, bevor sie in die Produktionsumgebung übertragen werden.

  • [ ] Entwerfen Sie eine Rollback-Lösung für die Bereitstellung.
