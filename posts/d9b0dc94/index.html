Sicherheitsdesign Checkliste

2017/09/30

This article was published before 377 days, and the content may be outdated.

ein, die Eingabevalidierung

  • Wenn die nicht vertrauenswürdigen Daten über Vertrauensgrenzen hinweg weitergegeben Prüfung (Politik prüft die Rechtmäßigkeit der Daten, einschließlich Whitelisting Mechanismus, etc.) Format-String, haben noch die Legitimität der Benutzereingabe zu verifizieren, kann das System eine Denial-of-Service oder die Offenlegung von Informationen zu vermeiden

  • Transfer Verboten nicht vertrauenswürdigen Java Runtime.exec () -Methode, die Rohdaten (wenn die Parameter Leerzeichen enthalten, doppelte Anführungszeichen, beginnend mit - oder / Schaltersymbol zeigt an, wenn ein Parameter, kann der Parameter Injection-Schwachstelle verursachen), ist es empfehlenswert, wenn JVM ausführen können externe Befehle deaktiviert werden, werden unbekannte Schwachstellen erheblich die Gefahr verringern, kann erheblich die Sicherheit der JVM verbessern.

  • es sollte zunächst auf den tatsächlichen Pfad normalisiert werden (spezielle Dateinamen, wie „..“, symbolische Links, harte Links, Verknüpfungen), bevor die Überprüfung Pfad

  • Von ZipInputStream Dateien in einem Verzeichnis extrahiert nicht im Programm, wenn das Programm erwartet wird, sollte es ablehnen, um zu extrahieren, oder es an einen sicheren Ort extrahieren

  • Von ZipInputStream Dateien extrahieren, wenn die Dateigröße nach der Dekompression eine bestimmte Grenze überschreitet, müssen Sie ablehnen unzip

  • Vor der Behandlung sicherzustellen, dass alle Daten vom Client, einschließlich: alle Parameter, URL, HTTP-Header-Informationen (zum Beispiel: Cookie-Name und Datenwert), einschließlich der Post von JavaScript, Flash oder anderer eingebetteter Code Rückinformation Bestimmung

  • Wenn eine potenziell gefährliche Zeichen müssen als Eingabe verwendet werden, stellen Sie sicher, dass Sie zusätzliche Sicherheitskontrollen durchführen, wie zum Beispiel: die Flucht Ein- und Ausgabecodierung, wie spezifische Sicherheits-API. Einige gemeinsamen gefährliche Zeichen, einschließlich, aber nicht beschränkt auf: <> “ ‘% () & + \ \’ \ ‘

  • Wenn die Standard-Gültigkeitsregel verwenden, können die folgenden Eingaben nicht überprüfen, müssen sie unabhängig überprüft, wie beispielsweise die Validierung null (00%) werden; Neue-Zeile (% 0D,% 0a, \ r, \ n) Verifizieren; Verifikationspfad Alternativ Zeichen “Punkt - Punkt - slash” (oder ../ .. ), wenn erweiterte Unterstützung UTF-8-Zeichensatzcodierung, Validierung Ersatzzeichen:% c0% ae% c0% ae / (unter Verwendung von Standard for Dual-Codierung oder andere Arten von Codierung)

  • starke Authentisierungsdaten aus der Eingabeumleitung (ein Angreifer kann direkt an den bösartigen Code übermittelt werden, um das Ziel umgeleitet werden, wodurch die Anwendungslogik zu vermeiden und Validierung vor allen Umleitungen durchgeführt wird)

  • Überprüfen Sie den Datentyp

  • Authentifizierungsdatenbereich

  • Authentifizierungsdatenlänge

Zweitens ist die Ausgabe des Encoders

  • ein Standardverfahren für jeden Ausgangscode verwendet wird, hat eine Regel getestet

  • codiert die von dem Server zurückgegebenen Daten an alle Client semantische Ausgangskodierung. Coding wie HTML, URL-Codierung erforderlich codierte Form je nach Szenario Auswahl bestimmte Anwendung

  • es sei denn, das Ziel Compiler sicher ist, kodieren sonst alle Zeichen

  • Ausgabe für SQL, XML und LDAP-Abfragen, semantische purify alle nicht vertrauenswürdige Daten

  • Für den Systembefehl arbeitet, reinigen alle Ausgaben nicht vertrauenswürdige Daten

Ausnahmebehandlung

  • die Offenlegung von sensiblen Informationen in Ausnahme verbieten (der Bereich von sensiblen Daten sollte basierend auf dem Ergebnis von Szenarien und Bedrohungsanalyse eines typischen Produktes sensiblerer Daten einschließlich Passwörtern, Kontonummern, persönliche Informationen, Kommunikationsaufzeichnungen, Schlüssel bestimmt werden, etc.)

  • verbieten Offenlegung der Anwendungsserver Ausnahme von Fingerabdruckinformationen (wie zB Version, Pfad, Architektur)

  • das Objekt in einen Zustand, bevor das Verfahren wieder hergestellt werden, wenn eine Abnormalität auftritt (wenn ein Betriebsfehler, Operationen Rollback, oder zu vermeiden, um den Zustand des Objekts zu ändern, das Objekt Zustand Konsistenz aufrechtzuerhalten)

  • I / O-Operationen - temporäre Dateien nach Gebrauch gelöscht werden sollen

  • nicht eingekapselten Daten Buffer-Objekte mit nicht vertrauenswürdigem Code ausgesetzt

  • Geben Sie die entsprechenden Zugriffsberechtigungen, wenn eine Datei in einem Mehrbenutzersystem zu schaffen nicht autorisierten Zugriff auf Dateien zu verhindern,

  • Wenn ein externer Prozessstrom externer Ausgang oder Fehlerinformationen durch seinen Ausgang seines Ausgangsstrom schnell entleert werden muß, um die Ausgabestrom-Puffer aufgebraucht ist und führen zu externem Prozess zu verhindern, blockiert.

  • Weiße Liste Dateiberechtigungen auf die freigegebenen Verzeichnisoperationen zu steuern, wie Lese- / Schreib- / Ausführungsberechtigungen

Drittens ist die Betriebsumgebung

  • Verwenden Sie keine gefährliche Kombination Lizenz und Ziele (wie nicht AllPermission der Erlaubnis gegeben, nicht vertrauenswürdigen Code nicht ReflectPermission Lizenz Zielkombinationen und suppressAccessChecks verwenden, nicht java.lang.RuntimePermission Lizenz Ziel in Kombination mit create)

  • Setzen Sie die JVM-Bytecode-Verifikation nicht deaktivieren, wenn Bytecode-Klassendateien als bösartig manipuliert, wird es ein Sicherheitsrisiko sein

  • Plattform Überwachung wird empfohlen, nicht Internet zu öffnen, wird der Zugriff auf die Umwelt innerhalb des Netzwerks beschränkt, wenn es eine Remote-Ausführung Verwundbarkeit Monitoring-Plattform ist, ein Sicherheitsrisiko für die Anwendung überwacht darstellen wird wird

  • Es wird empfohlen, dass alle sicherheitsrelevanten Codes (zum Beispiel Zugangskontrolle oder ein Benutzername und ein Kennwort Bestätigungscode) in einem JAR-Paket platziert

  • Produktionscode enthält keine Code oder Debug-Schnittstellen

Viertens Authentifizierung

  • Zusätzlich zu den spezifischen zu „öffnen“ Inhalt, alle Seiten und Ressourcen Authentifizierung benötigen, und Authentifizierungsfunktionen entwickelt, zu korrigieren

  • Alle von dem Authentifizierungsprozess muss auf dem Back-End-Server ausgeführt werden

  • Wo immer möglich, die Umstände, die Einrichtung und die Verwendung von Standard validiert Dienste (wie C4A) durch die Identität der Sicherheitstests

  • alle Authentifizierungs steuern sicher nicht erfolgreichen Authentisierung angeordnet sein sollte, wie beispielsweise einen Fehler fuzzy Aufforderung geben, sensible Informationen verbergen

  • Maßnahmen Gewalt Protokolleintrag zu verhindern, sollten die Bibliothek getroffen haben und erraten erraten (Batch-Login-Versuche das Passwort Wörterbuch mit durchgesickert sind) Lösung, überschreitet eine festgelegte Anzahl von fehlgeschlagenen Notwendigkeit Sperrcode oder Bilder Direktzugriffsbeschränkungen zu ermöglichen,

  • mit https-POST-Anfrage Übertragungen von Authentifizierungsanmeldeinformationen

  • Authentifizierung Fehlermeldung Fuzzy-Verarbeitung, wie die Verwendung von „Benutzername oder Kennwort Fehlern“ anstelle von „falschen Benutzername“ oder „Passwort“ eindeutig aufgefordert.

  • sensible Informationen oder externe Systemverbindung sollten funktionieren Authentifizierung konfiguriert werden, und eine gültige Authentifizierungssteuer

  • bei der Umsetzung der wichtigsten Operationen (wie die persönlichen Daten Passwort ändern Betrieb), reagiert wieder auf Benutzeridentitätsprüfung

  • die Verwendung von hochsensiblen oder wichtigen Handelskonto Multi-Faktor-Authentifizierungsmechanismen, wie Zahlungs Passwort, SMS Bestätigungscodes

7, Passwort-Management

  • die Verwendung von privatem oder schwachen Verschlüsselungsalgorithmus zu verbieten (wie das Verbot der Verwendung von DEM, SHA1, usw., ist es empfehlenswert, AES zu verwenden: 128 Wei, RSA: 2048 Wei, DSA: 2048 Bit)

  • Hashing-Algorithmen und auf Basis Wert Salz (Salz) sichere Speichermodus Kennwortinformationen

  • Passwort-Eingabefeld kann so ausgelegt werden, um zu zeigen, und Passwort Schalt verstecken

  • Passwort-Reset und ändern, müssen wir den rechtlichen Status von Sekundär überprüfen

  • Wenn das Passwort zurückgesetzt, Registrierung Telefonnummer und E-Mail-Antwort auf eine wirksame Überprüfung Link kann nur auf einen vorher registrierte E-Mail-Adresse oder Telefonnummer vorab gebündelt gesendet werden

  • vorläufiges Passwort und ein Link sollen für einen kurzen Zeitraum ausgelegt sein (beispielsweise 5 Minuten), Brute-Force zu verhindern

  • Wenn das Passwort zurückgesetzt, sollte ich SMS-Benachrichtigung, wenn der Benutzer in Betrieb ist, sagte ein Sicherheitsrisiko

  • Passwortkomplexität Einstellungen: Es wird empfohlen, mehr als acht Zeichen, einschließlich Buchstaben, Zahlen und Sonderzeichen usw.

  • Passwort-Szene Komplexität sollte die Kennwortüberprüfung

  • Passwort und können sich nicht anmelden auf der Konsole

  • Verbindungskonfigurationsdatenbank speichern das Kennwort des Benutzers in verschlüsselter Form

  • das vorgeschlagene Design-Modifikationen Passwort in regelmäßigen Abständen daran erinnert Mechanismus

8, Sitzungssicherheit

  • Bereinigen Sie die Sitzung und die zugehörigen Anmeldeinformationen unmittelbar nach dem Abmelden des Benutzers

  • Die Abmeldefunktion sollte die zugehörige Sitzung oder Verbindung vollständig beenden

  • Cookie Erhöhung der Sicherheit und fügt hinzu, „Httponly“ und „sicher“ Attribut (bei „sichere“ Eigenschaft auf true gesetzt ist, ein Cookie erstellen wird in einer sicheren Form an den Server übertragen werden, das heißt, nur HTTPS-Verbindungen sind Browser übermittelte Sitzung an den Server zur Authentifizierung wird die HTTP-Verbindung zu den Informationen nicht überträgt, ist es kein Problem gestohlen Plätzchen; set „Httponly“ -Attribut, durch ein Programm (JS-Skript, das Applet, etc.) kann nicht gelesen werden Cookie-Informationen, die auch das Risiko von XSS-Cross-Site-Scripting-Angriffen reduzieren.

  • Session-Cookies sollten für den Ablauf ausgelegt sein und sofort nach dem Timeout ablaufen

  • Wenn das Design es Benutzern ermöglicht, sich gleichzeitig an einem Mehrkanal-Terminal anzumelden, empfiehlt es sich, allgemeine Anmeldebeschränkungen für Geräte zu treffen.

  • Legen Sie die Domäne und den Pfad für das Cookie fest, das die authentifizierte Sitzungskennung enthält, und legen Sie einen entsprechenden Grenzwert für die Site fest. Die Standard-Cookie-Domäne ist der aktuelle Domänenname und der Standard-Cookie-Pfad ist der Verzeichnispfad der aktuellen Seite. Wenn Sie domänenübergreifend oder in anderen Pfaden auf Cookies zugreifen möchten, müssen Sie diese beiden Eigenschaften Domain und Pfad zurücksetzen.

  • Die Abmeldefunktion sollte für alle authentifizierten Seiten verfügbar sein

  • Legen Sie einen Sitzungszeitlimit fest, der so kurz wie möglich ist. Berücksichtigen Sie dabei die Risiko- und Geschäftsfunktionsanforderungen. Normalerweise sollte es ein paar Stunden nicht überschreiten.

  • Nicht Sitzungsidentifikatoren in der URL belichten, oder melden Sie sich die Fehlermeldung sollte die Sitzungskennung erscheinen nur in der http-Header-Informationen nicht auf die Sitzungskennung GET-Parameter übergeben

  • periodisch erzeugt eine neue Session-Kennung und nicht periodisch einen Sitzungsidentifizierer machen (session insbesondere solche, die die ursprüngliche Kennung lindern können, sind im Fall von hijacking) erhaltene

  • Wenn bei der Authentifizierung die Verbindung von HTTP zu HTTPS wechselt, wird eine neue Sitzungs-ID generiert. In der Anwendung bei Eingriffen, entführte die empfohlene Dauer Verwendung HTTPS, Sie nicht hin und her zwischen dem HTTPS und HTTP umgeschaltet werden sollen, wird die Sitzung Übergabeprozess wirksam zu verhindern.

  • Request Forgery Sicherheitsanfälligkeit als Standard von Operationen auf der Server-Seite eines sicheren Session-Managements durchgeführt, die Umsetzung von Recht Authentifizierung und Zugriffskontrolle für jede Sitzung, zu verhindern, dass Cross-Site-Präsenz von CSRF

9, Zugangskontrolle

  • Separate privilegierte Logik von anderem Anwendungscode

  • Beschränken Sie den Zugriff auf Dateiressourcen nur auf autorisierte Benutzer

  • Beschränken Sie nur autorisierten Benutzern den Zugriff auf geschützte URLs

  • Beschränken Sie den Zugriff auf geschützte Funktionen oder Dienste nur auf autorisierte Benutzer

  • Es wird empfohlen, dass nur autorisierte Benutzer auf direkte Objektreferenzen zugreifen können

  • Beschränken Sie den Zugriff auf geschützte Anwendungsdaten nur auf autorisierte Benutzer

  • Beschränken Sie den Zugriff auf sicherheitsrelevante Konfigurationsinformationen nur auf autorisierte Benutzer

  • Beschränken Sie den Zugriff auf geschützte lokale Programme oder Ressourcen nur auf autorisierte externe Anwendungen oder Schnittstellen

  • Zugriffskontrollregeln, die auf der Serverseite erzwungen werden, und Zugriffskontrollregeln, die am Frontend implementiert sind, müssen übereinstimmen

  • Geben Sie beim Erstellen von Dateien auf dem Server die entsprechenden Zugriffsrechte (Lesen / Schreiben / Ausführen) an

  • Beim Zurücksetzen Berechtigungen geändert werden, sollte das Protokoll die gute und SMS-Benachrichtigung aufzeichnen, wenn der Benutzer selbst in Betrieb ist, sagte ein Sicherheitsrisiko bestehen kann

10, Protokollspezifikation

  • Speichern Sie keine vertraulichen Informationen im Protokoll, einschließlich System-Fingerabdruckinformationen, Sitzungs-ID, Kontopasswort, ID, ID usw.

  • Stellen Sie sicher, dass der Protokollsatz wichtige Protokollereignisdaten enthält

  • Notieren Sie alle fehlgeschlagenen und erfolgreichen Eingabevalidierungen

  • Notieren Sie alle fehlgeschlagenen und erfolgreichen Authentifizierungsdatensätze

  • Notieren Sie alle fehlgeschlagenen und erfolgreichen Zugriffs- und Vorgangsdatensätze

  • Erfassen Sie offensichtliche Änderungsereignisse, einschließlich Änderungen an Statusdaten

  • Aufzeichnung eines Sitzungstokenversuchs mit einer ungültigen Verbindung oder abgelaufen

  • Zeichnen Sie alle Betriebsverhalten der Verwaltungsfunktionen auf, einschließlich, aber nicht beschränkt auf Änderungen der Sicherheitskonfigurationseinstellungen

  • Notieren Sie alle fehlgeschlagenen und erfolgreichen Back-End-Verbindungen

  • Notieren Sie die Fehlerinformationen des Verschlüsselungsmoduls

  • Es ist verboten, das Protokoll direkt im WEB-Verzeichnis zu speichern, auf das der Browser zugreifen kann.

11, sensible Informationen

  • Temporäre Erzeugung sensibler Daten (in Speicher oder Dateien geschrieben) mit zeitnahen Entfernungs- und Freigabemechanismen

  • Geben Sie keine sensiblen Informationen in die HTTP-GET-Anforderungsparameter wie Benutzername, Passwort, Kartennummer, ID usw. ein.

  • Deaktivieren Sie das automatische Ausfüllen des Formulars, da das Formular möglicherweise vertrauliche Informationen enthält, einschließlich Authentifizierungsinformationen

  • Speichern Sie keine Passwörter oder andere vertrauliche Informationen in Klartext auf dem Client

  • SSL-verschlüsselte Übertragung für alle sensiblen Informationen

  • Codieren Sie keine sensiblen Informationen (einschließlich Verschlüsselungsschlüssel usw.) im Programm

  • Es ist verboten, vertrauliche Informationen wie Passwort, ID-Nummer, Bankkartennummer und Karteninhabername im Klartext zu speichern.

  • Speichern Sie keine vertraulichen Informationen im Protokoll, einschließlich, aber nicht beschränkt auf Systemdetails, Sitzungskennungen, Passwörter usw.

  • Es ist verboten, die Fingerabdruckinformationen des Anwendungsservers in der Ausnahme offenzulegen, wie z. B. Version, Pfad, Komponentenversion usw.

  • Es ist verboten Quellcode oder Sql auf Open Source Plattformen oder Communities hochzuladen, wie zB github, Open Source China, etc.

  • Anforderung enthält sensitive Parameter (wie Auftragsnummer, ID, etc.) sind verworren werden, verhindern, dass die Gefahr des Entstehens Verfahrinformation Parameter

  • Wenn vertrauliche Informationen auf einer Webseite angezeigt werden sollen, sollten sensible Felder im Hintergrund desensibilisiert werden.

  • Rücksendungsdaten sollten keine anderen Geschäftsdaten als Anfragen enthalten, insbesondere keine sensiblen Informationsdaten

12, Passwortwiederherstellungssicherheit

  • Server-Seite, um Authentifizierung zu tun, vermeiden Sie, Front-End-Kontrolle zu umgehen

  • Fügen Sie einen sekundären Authentifizierungsfaktor hinzu, z. B. einen Bestätigungscode

  • Einbeziehen von Authentifizierungs-Token für die Authentifizierung usw. geben den Verifizierungsinhalt nicht direkt an den Benutzer zurück.

  • Verschlüsselung der Authentifizierungsinformationen, starker Algorithmus empfohlen (AES wird empfohlen: 128 Bit, RSA: 2048 Bit, DSA: 2048 Bit)

  • Die Parameter im Authentifizierungszertifikat sollten verwechselt werden

  • Bei mehreren Verifizierungsvorgängen wird jeder Überprüfungsmechanismus sortiert, wenn ein Sicherheitsrisiko besteht, das den vorherigen Überprüfungsmechanismus überspringt und direkt zum letzten Schritt der Authentifizierung führt.

  • Überprüfung des Handy-Kurznachrichtencodes, es ist notwendig zu überprüfen, ob die Mobiltelefonnummer und die Kurzmitteilung übereinstimmen

  • Das Eingabefeld sollte die Rechtmäßigkeit der Eingabedaten überprüfen, um Cross-Site-Scripting-Angriffe durch XSS zu verhindern

  • Password Recovery Link begrenzt die effektive Zugriffszeit und die Anzahl der Wiederverwendungen (nicht wiederverwendbar)

Dreizehn, SQL-Injektion

  • Traue niemals Benutzereingaben, überprüfe alle Benutzereingaben, schließe das Filtern und Entkommen von SQL-Anweisungen ein

  • Verwenden Sie niemals dynamisches Assembly-SQL, Sie können parametrisiertes SQL verwenden oder gespeicherte Prozeduren für den Datenabfragezugriff verwenden

  • Verwenden Sie niemals Administratorrechte für Datenbankverbindungen, verwenden Sie separate nicht privilegierte Berechtigungen für jede Anwendung und konfigurieren Sie eine begrenzte Anzahl von Datenbankverbindungen

  • Speichern Sie keine vertraulichen Informationen im Klartext, verwenden Sie sensible Methoden wie Verschlüsselung oder Hashing oder Verwirrung, um sensible Informationen zu desensibilisieren.

  • Ausnahmeinformationen Anwendungen mit sensiblen Daten sollten nicht minimal Hinweise gegeben werden, empfiehlt die Fehlermeldung in der Originalverpackung Fehler anpassen, kann die Ausnahmeinformationen in einer separaten Datenbank-Tabelle gespeichert werden

  • XML-Injektion - Zusammensetzen von XML mit String / StringBuffer / StringBuilder / StringFormat nicht

  • Es wird empfohlen, das Attribut oder den Inhalt des XML-Elements zu umgehen

Vierzehn, XSS-Cross-Site-Scripting-Angriff

  • Eingabedaten filtern und entkommen, einschließlich, aber nicht beschränkt auf <> “% () & + \ \” und andere gefährliche Sonderzeichen

  • HTML-Entweichen von Daten, wenn Daten zu HTML-Elementattributen oder Inhalt hinzugefügt werden

  • Script-Escaping-Daten, wenn es zu einem Skript hinzugefügt wird

  • CSS-Escape von Daten, wenn Daten zum Stil hinzugefügt werden

15, CSRF Cross-Site-Anfrage Fälschung

  • Es wird empfohlen, die CSRF-Token-Verifizierung in jedem Schlüsselformular einzuführen (zufällige Zeichenfolge, die in der Sitzung generiert wurde, Prüfung nach der Einreichung)

  • Benutzer müssen beim Senden von Schlüsselformularen eine sekundäre Authentifizierung durchführen (Passwort eingeben, KEY-Taste drücken, Bildverifizierungscode eingeben, SMS-Bestätigungscode)

  • Überprüfen Sie den Request Referer (z. B. domänenübergreifende, systeminterne Anwendung)

16, Datei-Upload-Sicherheit

  • Der Upload-Vorgang sollte einen Authentifizierungsmechanismus entwickeln und eine rechtliche Identitätsprüfung durchführen

  • Laden Sie nur relevante Dokumenttypen hoch, die den geschäftlichen Anforderungen entsprechen

  • indem Sie die Datei-Header-Informationen, wie zum Beispiel JPEG (jpg) Datei-Header-Informationen (hexadezimal) Prüfung: FFD8FF, stellen Sie sicher, dass ein Dokumenttyp erwartet wird

  • Verwenden Sie keine Dateien auf die gleiche Web-Anwendungsumgebung speichern, wird empfohlen, die Datei in einem eigenen Dateiserver in einer separaten Domain-Name-Server-Konfigurationsdatei speichern besser zugreifen

  • Beschränken Sie das Hochladen von Dateien, die vom Webserver analysiert werden können, wie zB jsp, php usw.

  • Upload-Dateien werden in binärer Form heruntergeladen Es wird empfohlen, keinen direkten Zugriff zu gewähren (um zu verhindern, dass Trojaner-Dateien direkt ausgeführt werden)

  • Erteilen Sie dem freigegebenen Dateispeicherverzeichnis keine ausführbaren Berechtigungen

  • Deaktivieren Sie clientseitige Upload- / Download-Pfade für benutzerdefinierte Dateien (z. B. zum Springen verwenden Sie ../../../../).

  • Nach Datei-Upload umbenennen (müssen Namensregeln entsprechend den tatsächlichen Geschäftsanforderungen entwickeln)

17, Komponentensicherheit

  • Bei Verwendung von Zufallszahlenfunktionen (z. B. java.security.SecureRandom-Klasse) werden starke Zufallszahlenfunktionen empfohlen

  • Minimieren Sie nicht benötigte Funktionen und Methoden in Komponenten, um unbekannte Sicherheitsrisiken zu vermeiden

  • Setzen Sie die im System verwendeten Sperrobjekte nicht dem nicht vertrauenswürdigen Code aus

  • Es wird empfohlen, SSL anstelle von Socket für sichere Dateninteraktion zu verwenden

  • Lokale Methodenaufrufe einkapseln (alle lokalen Methoden sollten als privat definiert sein und dann nur über eine Wrapper-Methode aufgerufen werden)

  • Verwenden Sie einen Sicherheitsmanager wie java.security oder Sicherheitskomponenten von Drittanbietern, um sensible Vorgänge zu schützen

  • Wenn Sie benutzerdefinierte Class Loader schreiben müssen abdecken GetPermissions () Funktion, vor allen Vertriebsrechte am Quellcode wird, sollte die übergeordnete Klasse super.getPermissions () aufrufen, Funktion, zusätzlich eine benutzerdefinierte Richtlinie, systemweite Standardsicherheitsrichtlinie auch umzusetzen ist Anwendung.

  • Vermeiden Sie verlassen sich voll und ganz auf die Standard automatische Signatur und Authentifizierungsmechanismen URLClassLoader java.util.jar vorgesehen ist, sollte eine Zertifikatskette Code Quelle (-Code-Source) erhalten Klassen zu laden, dann überprüfen Sie, ob das Zertifikat auf den lokalen Schlüsselspeicher gehört (der Schlüsselspeicher) in vertrauenswürdiger Unterzeichner

18, Schnittstellen-Sicherheit

  • IP-Steuerung der Anruferquelle, z. B. Firewall, Host-Verweigerung, Nginx-Verweigerung usw.

  • Authentifizierung der Identität des Anrufers, z. B. Schlüssel, Geheimnis, Zertifikat usw.

  • Aufruf der Parameterauthentifizierung, Entwurf eines Parameterfehlertoleranzmechanismus, um das Durchlaufen von Parametern für sensible Datensicherheitsprobleme zu vermeiden

  • Verwenden Sie die digitale Signatur, um sicherzustellen, dass die Identität der Schnittstelle vertrauenswürdig ist und die Daten manipulationssicher sind

  • Einstellungen für die Steuerung der Anruferberechtigung

  • Häufigkeit und Gültigkeitszeitraum für die Kontrolle aufrufen

  • Call-Verhalten Echtzeit-Erkennung, Sperren von Ausnahmen

  • Idempotenzprüfung, um die Datenkonsistenz zu erhalten

  • Application Access Security Gateway für APPID / KEY Identitätsauthentifizierung, verschlüsselte Übertragung und Digest-Signatur-Sicherheit

19, ruft Dubbo Sicherheit

  • die Verwendung von Token-Verifizierungszugangskontrolle, für die Verbraucher verhindern, dass unter Umgehung Zugriff auf die Registry-Provider, in der Registry Kontrollbehörde zu entscheiden, ob das Token an die Verbraucher abgegeben werden

  • Adopt Filter IP-Whitelist-Zugangskontrolle, und auch Dubbo unordentlichen Anruf zwischen Produktionssystem und Testsystem zu verhindern

  • Überprüfen Sie bei Bedarf (z. B. bei vertraulichen Informationen) den Benutzernamen und das Passwort, wenn Sie eine Verbindung zum Registrierungscenter Dubbo herstellen

20, Redis ruft Sicherheit auf

  • Die Überprüfung der Client-IP-Zugriffssteuerung sollte aktiviert sein

  • Die Client-Authentifizierung sollte aktiviert sein

  • Sensible Informationen sollten nicht in Klartext in Redis gespeichert werden
